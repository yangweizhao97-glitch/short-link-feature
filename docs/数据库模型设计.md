# 短链接功能数据库模型设计

## 设计目标

1. 支持短码全局唯一。
2. 支持短链接访问统计。
3. 支持短链接过期时间。
4. 支持软禁用，不物理删除核心记录。
5. 为后续扩展预留空间，例如自定义短码、访问日志、来源分析、风控、管理员审核。

## 设计假设

1. 创建短链接需要登录，短链接归属于创建用户。
2. 短链接默认永久有效，`expires_at = null` 表示不过期。
3. MVP 阶段至少记录总点击数。
4. 访问明细日志作为可选表设计，便于后续统计扩展。
5. 数据库以 PostgreSQL 作为参考，其他关系型数据库可按等价类型调整。

## 实体关系

```text
users
  1
  |
  | user_id
  *
short_links
  1
  |
  | short_link_id
  *
short_link_clicks
```

说明：

1. 一个用户可以创建多个短链接。
2. 一个短链接可以产生多条访问记录。
3. MVP 可以只使用 `short_links.click_count`；需要详细分析时再启用 `short_link_clicks`。

## 表一：short_links

短链接主表，保存短码、原始 URL、状态、过期时间和聚合访问次数。

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|---|---|---:|---|---|
| `id` | uuid / varchar | 是 | 系统生成 | 短链接主键 |
| `user_id` | uuid / varchar | 是 | 无 | 创建用户 ID |
| `original_url` | text | 是 | 无 | 用户提交的原始长链接 |
| `normalized_url` | text | 是 | 无 | 归一化 URL，用于重复判断 |
| `short_code` | varchar(32) | 是 | 无 | 短码，全局唯一 |
| `status` | varchar(20) | 是 | `active` | 短链接状态 |
| `click_count` | bigint | 是 | `0` | 成功跳转次数 |
| `expires_at` | timestamptz | 否 | `null` | 过期时间，null 表示不过期 |
| `last_clicked_at` | timestamptz | 否 | `null` | 最近一次成功访问时间 |
| `created_at` | timestamptz | 是 | 当前时间 | 创建时间 |
| `updated_at` | timestamptz | 是 | 当前时间 | 更新时间 |

### 字段约束

1. `short_code` 必须全局唯一。
2. `status` 只允许 `active`、`disabled`。
3. `click_count` 必须大于等于 `0`。
4. `original_url` 和 `normalized_url` 不允许为空字符串。
5. `expires_at` 允许为空；为空表示永久有效。
6. `user_id` 应关联用户表主键。

### 推荐索引

| 索引 | 类型 | 说明 |
|---|---|---|
| `uk_short_links_short_code` | unique | 保证短码全局唯一 |
| `idx_short_links_user_created_at` | btree | 支持用户短链接列表按创建时间倒序查询 |
| `idx_short_links_user_status_created_at` | btree | 支持按用户和状态筛选 |
| `idx_short_links_expires_at` | btree | 支持过期清理或过期扫描 |
| `idx_short_links_normalized_url` | btree | 支持重复 URL 判断 |

### 重复 URL 复用约束

如果业务要求“同一用户重复提交同一个长链接时返回已有 active 短链接”，可以采用以下策略之一：

1. 应用层查询 `user_id + normalized_url + status = active`，命中则返回已有记录。
2. 数据库增加部分唯一索引，防止并发重复创建 active 记录。

PostgreSQL 参考：

```sql
CREATE UNIQUE INDEX uk_short_links_user_active_normalized_url
ON short_links (user_id, normalized_url)
WHERE status = 'active';
```

说明：

1. 该索引只限制 active 记录。
2. 禁用后的旧记录不会阻止用户重新创建同一 URL。
3. 如果产品希望“禁用后仍复用旧记录”，则不要使用该部分唯一索引，应由业务逻辑控制。

## 表二：short_link_clicks

访问明细表，用于后续分析访问来源、趋势、设备信息和安全审计。MVP 如果只需要总点击数，可以暂缓落地该表。

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|---|---|---:|---|---|
| `id` | uuid / varchar | 是 | 系统生成 | 点击记录主键 |
| `short_link_id` | uuid / varchar | 是 | 无 | 短链接 ID |
| `clicked_at` | timestamptz | 是 | 当前时间 | 点击时间 |
| `ip_hash` | varchar(128) | 否 | `null` | 脱敏后的 IP 哈希 |
| `user_agent` | text | 否 | `null` | 浏览器或客户端 UA |
| `referer` | text | 否 | `null` | 来源页面 |
| `country_code` | varchar(8) | 否 | `null` | 后续扩展：国家或地区 |
| `device_type` | varchar(32) | 否 | `null` | 后续扩展：设备类型 |
| `created_at` | timestamptz | 是 | 当前时间 | 记录创建时间 |

### 字段约束

1. `short_link_id` 应关联 `short_links.id`。
2. `ip_hash` 不保存明文 IP。
3. `user_agent` 和 `referer` 需要限制最大长度，避免日志表被超长输入撑爆。
4. `country_code`、`device_type` 属于扩展字段，MVP 可以为空。

### 推荐索引

| 索引 | 类型 | 说明 |
|---|---|---|
| `idx_short_link_clicks_link_clicked_at` | btree | 支持按短链接查询访问趋势 |
| `idx_short_link_clicks_clicked_at` | btree | 支持按时间范围统计或清理 |
| `idx_short_link_clicks_ip_hash` | btree | 支持基础风控排查，可选 |

## 状态枚举

| 值 | 说明 | 是否可跳转 |
|---|---|---:|
| `active` | 正常可用 | 是 |
| `disabled` | 用户或管理员禁用 | 否 |

后续如需要审核能力，可以扩展：

| 值 | 说明 |
|---|---|
| `pending_review` | 等待审核 |
| `blocked` | 因风控或安全原因封禁 |

新增状态时应同步更新 API 文档、业务逻辑和验收测试。

## 过期时间设计

1. `expires_at = null` 表示永久有效。
2. `expires_at <= now()` 表示已过期。
3. 过期记录不需要立即删除；访问时实时判断即可。
4. 可后续增加定时任务，将长期过期记录归档或标记为不可见。
5. 列表和详情接口仍可展示已过期短链接，便于用户理解为什么无法跳转。

## 访问统计设计

### MVP 方案

只维护 `short_links.click_count` 和 `last_clicked_at`。

成功跳转时执行原子更新：

```sql
UPDATE short_links
SET
  click_count = click_count + 1,
  last_clicked_at = now(),
  updated_at = now()
WHERE id = $1;
```

优点：

1. 实现简单。
2. 查询详情和列表时无需聚合。
3. 能满足基础点击次数展示。

限制：

1. 无法分析访问来源。
2. 无法按时间统计趋势。
3. 无法做去重访客统计。

### 扩展方案

启用 `short_link_clicks` 记录每次访问明细，同时继续维护 `click_count` 作为聚合缓存。

建议：

1. 写点击明细失败时，不应影响跳转主流程。
2. `click_count` 使用原子递增，避免并发访问丢失计数。
3. 访问日志可设置保留周期，避免表无限增长。

## 安全和隐私约束

1. 不保存明文 IP，使用带盐哈希后的 `ip_hash`。
2. 不在访问日志中保存登录凭证、Cookie、Token 或完整请求头。
3. `referer` 和 `user_agent` 属于外部输入，写入前应限制长度。
4. API 响应不返回其他用户的短链接数据。
5. 所有数据库查询必须参数化，禁止拼接 SQL。
6. 用户提交的 URL 必须在 API 边界校验，不依赖数据库约束兜底。

## 后续扩展字段建议

### short_links 可扩展字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `custom_alias` | varchar(64) nullable | 用户自定义短码或别名 |
| `description` | varchar(255) nullable | 用户备注 |
| `created_by_type` | varchar(32) nullable | 区分用户、管理员、系统创建 |
| `max_clicks` | bigint nullable | 最大访问次数 |
| `blocked_reason` | varchar(255) nullable | 风控或管理员封禁原因 |
| `metadata` | jsonb nullable | 少量扩展信息 |

### short_link_clicks 可扩展字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `browser` | varchar(64) nullable | 浏览器 |
| `os` | varchar(64) nullable | 操作系统 |
| `utm_source` | varchar(128) nullable | 营销来源 |
| `utm_campaign` | varchar(128) nullable | 营销活动 |
| `is_bot` | boolean nullable | 是否疑似机器人 |

说明：

1. 通用扩展字段不要过早滥用。
2. 高频筛选字段应独立成列，不建议长期藏在 `metadata` 中。
3. `metadata` 适合低频、非关键查询信息。

## 参考 DDL

以下 DDL 仅作为 PostgreSQL 参考，具体实现应按项目 ORM 或迁移工具调整。

```sql
CREATE TABLE short_links (
  id uuid PRIMARY KEY,
  user_id uuid NOT NULL,
  original_url text NOT NULL,
  normalized_url text NOT NULL,
  short_code varchar(32) NOT NULL,
  status varchar(20) NOT NULL DEFAULT 'active',
  click_count bigint NOT NULL DEFAULT 0,
  expires_at timestamptz NULL,
  last_clicked_at timestamptz NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  CONSTRAINT uk_short_links_short_code UNIQUE (short_code),
  CONSTRAINT ck_short_links_status CHECK (status IN ('active', 'disabled')),
  CONSTRAINT ck_short_links_click_count CHECK (click_count >= 0),
  CONSTRAINT ck_short_links_original_url_not_empty CHECK (length(trim(original_url)) > 0),
  CONSTRAINT ck_short_links_normalized_url_not_empty CHECK (length(trim(normalized_url)) > 0)
);

CREATE INDEX idx_short_links_user_created_at
ON short_links (user_id, created_at DESC);

CREATE INDEX idx_short_links_user_status_created_at
ON short_links (user_id, status, created_at DESC);

CREATE INDEX idx_short_links_expires_at
ON short_links (expires_at);

CREATE INDEX idx_short_links_normalized_url
ON short_links (normalized_url);

CREATE UNIQUE INDEX uk_short_links_user_active_normalized_url
ON short_links (user_id, normalized_url)
WHERE status = 'active';
```

```sql
CREATE TABLE short_link_clicks (
  id uuid PRIMARY KEY,
  short_link_id uuid NOT NULL,
  clicked_at timestamptz NOT NULL DEFAULT now(),
  ip_hash varchar(128) NULL,
  user_agent text NULL,
  referer text NULL,
  country_code varchar(8) NULL,
  device_type varchar(32) NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  CONSTRAINT fk_short_link_clicks_short_link
    FOREIGN KEY (short_link_id)
    REFERENCES short_links (id)
    ON DELETE CASCADE
);

CREATE INDEX idx_short_link_clicks_link_clicked_at
ON short_link_clicks (short_link_id, clicked_at DESC);

CREATE INDEX idx_short_link_clicks_clicked_at
ON short_link_clicks (clicked_at DESC);

CREATE INDEX idx_short_link_clicks_ip_hash
ON short_link_clicks (ip_hash);
```

## 验收标准

1. `short_links.short_code` 有唯一约束。
2. 同一用户重复提交同一 active URL 时不会产生重复 active 记录。
3. `click_count` 默认值为 `0`，并支持原子递增。
4. `expires_at = null` 表示不过期。
5. 已过期短链接可被识别并阻止跳转。
6. 已禁用短链接不物理删除。
7. 访问明细表支持后续按短链接和时间查询访问记录。
8. 访问日志不保存明文 IP、Cookie、Token 等敏感信息。
9. 数据模型支持后续扩展自定义短码、访问上限、风控状态和统计维度。
