# 短链接功能后端开发规格

## 当前假设

1. 创建短链接需要登录，短链接归属于创建用户。
2. 短码默认由系统生成，暂不支持用户自定义短码。
3. 同一用户重复提交同一个长链接时，默认返回已有可用短链接。
4. 短链接默认永久有效，但数据模型预留过期时间。
5. 访问短链接使用 `302` 临时重定向。
6. MVP 需要记录点击次数，不做复杂访问分析。
7. 只允许 `http` / `https` 原始链接，禁止 `javascript:`、`data:`、内网地址、localhost。
8. 短链接不存在、禁用或过期时返回错误页面；API 场景返回 JSON 错误。

## 功能目标

实现一个短链接服务，用户提交长链接后，系统生成一个更短的可访问链接。访问短链接时，系统根据短码查询原始链接，并重定向到原始地址。

核心目标：

1. 支持登录用户创建短链接。
2. 支持通过短码访问并跳转到原始链接。
3. 支持基础异常处理，包括非法 URL、短码不存在、短链禁用、短链过期。
4. 支持记录短链接点击次数。
5. 为后续扩展预留字段，例如过期时间、访问日志、自定义短码、统计分析。

非目标：

1. 暂不支持二维码生成。
2. 暂不支持用户自定义短码。
3. 暂不支持地理位置、设备类型、A/B 测试等定向跳转。
4. 暂不支持自定义短链域名。
5. 暂不接入第三方恶意链接检测服务。
6. 暂不开放第三方 API key 管理能力。

## 接口设计

### 创建短链接

`POST /api/short-links`

请求体：

```json
{
  "url": "https://example.com/articles/123"
}
```

成功响应：

```json
{
  "id": "sl_123",
  "originalUrl": "https://example.com/articles/123",
  "shortCode": "aB91xZ",
  "shortUrl": "https://s.example.com/aB91xZ",
  "expiresAt": null,
  "createdAt": "2026-06-14T10:00:00Z"
}
```

规则：

1. 用户必须已登录。
2. `url` 必填。
3. `url` 必须是合法 `http` 或 `https` URL。
4. 禁止跳转到内网地址、localhost、保留 IP 地址段。
5. 如果同一用户已经创建过相同 `originalUrl` 且记录仍可用，返回已有短链接。
6. 如果生成短码冲突，系统自动重新生成。

错误响应示例：

```json
{
  "code": "INVALID_URL",
  "message": "URL is invalid or unsupported."
}
```

### 查询短链接详情

`GET /api/short-links/{id}`

成功响应：

```json
{
  "id": "sl_123",
  "originalUrl": "https://example.com/articles/123",
  "shortCode": "aB91xZ",
  "shortUrl": "https://s.example.com/aB91xZ",
  "status": "active",
  "clickCount": 42,
  "expiresAt": null,
  "createdAt": "2026-06-14T10:00:00Z",
  "updatedAt": "2026-06-14T10:00:00Z"
}
```

规则：

1. 用户必须已登录。
2. 普通用户只能查询自己创建的短链接。
3. 管理员是否可查询所有短链接，作为后续权限扩展。

### 查询当前用户短链接列表

`GET /api/short-links?page=1&pageSize=20`

成功响应：

```json
{
  "items": [
    {
      "id": "sl_123",
      "originalUrl": "https://example.com/articles/123",
      "shortCode": "aB91xZ",
      "shortUrl": "https://s.example.com/aB91xZ",
      "status": "active",
      "clickCount": 42,
      "createdAt": "2026-06-14T10:00:00Z"
    }
  ],
  "page": 1,
  "pageSize": 20,
  "total": 1
}
```

规则：

1. 用户必须已登录。
2. 只返回当前用户创建的短链接。
3. 默认按创建时间倒序排序。

### 禁用短链接

`PATCH /api/short-links/{id}`

请求体：

```json
{
  "status": "disabled"
}
```

成功响应：

```json
{
  "id": "sl_123",
  "status": "disabled",
  "updatedAt": "2026-06-14T10:05:00Z"
}
```

规则：

1. 用户必须已登录。
2. 普通用户只能禁用自己创建的短链接。
3. 禁用后短链接不可跳转。
4. MVP 不做物理删除，使用软禁用。

### 访问短链接并跳转

`GET /{shortCode}`

行为：

1. 根据 `shortCode` 查询短链接。
2. 如果短链接存在、状态为 `active`、未过期，则记录点击次数并返回 `302 Redirect`。
3. `Location` 指向 `originalUrl`。
4. 如果短链接不存在、禁用或过期，返回错误页面或统一错误响应。

响应示例：

```http
HTTP/1.1 302 Found
Location: https://example.com/articles/123
```

## 数据模型

### `short_links`

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string / uuid | 短链接主键 |
| `user_id` | string / uuid | 创建用户 ID |
| `original_url` | text | 原始长链接 |
| `normalized_url` | text | 归一化后的 URL，用于重复判断 |
| `short_code` | string | 短码，唯一 |
| `status` | enum | `active` / `disabled` |
| `click_count` | integer | 点击次数 |
| `expires_at` | datetime nullable | 过期时间，默认 null |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

索引建议：

1. `unique(short_code)`
2. `index(user_id, created_at)`
3. `unique(user_id, normalized_url, status)` 或通过业务逻辑控制重复复用
4. `index(expires_at)`

### `short_link_clicks`，可选

MVP 如果只需要点击总数，可以暂不建访问日志表。若需要后续统计，建议增加：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string / uuid | 点击记录 ID |
| `short_link_id` | string / uuid | 短链接 ID |
| `ip_hash` | string nullable | 脱敏后的 IP |
| `user_agent` | text nullable | 访问 UA |
| `referer` | text nullable | 来源 |
| `clicked_at` | datetime | 点击时间 |

MVP 建议：

1. 第一阶段只更新 `click_count`。
2. 如果后续需要分析来源、设备、地区，再启用 `short_link_clicks`。

## 核心流程

### 创建短链接流程

1. 接收 `POST /api/short-links` 请求。
2. 校验用户登录状态。
3. 校验 `url` 是否存在。
4. 解析 URL。
5. 校验协议是否为 `http` 或 `https`。
6. 校验 host 不属于 localhost、内网地址或保留地址。
7. 对 URL 做归一化，得到 `normalized_url`。
8. 查询当前用户是否已有相同 `normalized_url` 的 active 短链接。
9. 如果存在，直接返回已有短链接。
10. 如果不存在，生成随机短码。
11. 检查短码是否冲突。
12. 如冲突，重新生成，直到成功或达到重试上限。
13. 写入 `short_links`。
14. 返回短链接信息。

### 访问跳转流程

1. 接收 `GET /{shortCode}` 请求。
2. 校验 `shortCode` 格式。
3. 根据 `shortCode` 查询短链接。
4. 如果不存在，返回 404。
5. 如果状态不是 `active`，返回 410 或错误页面。
6. 如果 `expires_at` 不为空且已过期，返回 410 或错误页面。
7. 增加 `click_count`。
8. 返回 `302 Redirect` 到 `original_url`。

### 禁用短链接流程

1. 接收 `PATCH /api/short-links/{id}` 请求。
2. 校验用户登录状态。
3. 查询短链接是否存在。
4. 校验当前用户是否有权限操作。
5. 更新 `status` 为 `disabled`。
6. 返回更新后的状态。

## 异常处理

### API 错误格式

统一返回：

```json
{
  "code": "ERROR_CODE",
  "message": "Human readable error message."
}
```

### 错误码建议

| 场景 | HTTP 状态码 | code |
|---|---:|---|
| 未登录 | 401 | `UNAUTHENTICATED` |
| 无权限访问他人短链 | 403 | `FORBIDDEN` |
| URL 为空 | 400 | `URL_REQUIRED` |
| URL 格式非法 | 400 | `INVALID_URL` |
| URL 协议不支持 | 400 | `UNSUPPORTED_URL_SCHEME` |
| URL 指向内网或 localhost | 400 | `URL_NOT_ALLOWED` |
| 短链接不存在 | 404 | `SHORT_LINK_NOT_FOUND` |
| 短链接已禁用 | 410 | `SHORT_LINK_DISABLED` |
| 短链接已过期 | 410 | `SHORT_LINK_EXPIRED` |
| 短码生成失败 | 500 | `SHORT_CODE_GENERATION_FAILED` |
| 系统内部错误 | 500 | `INTERNAL_ERROR` |

### 需要特别处理的异常场景

1. 长链接为空字符串。
2. 长链接不是 URL。
3. 长链接缺少协议。
4. 长链接协议为 `javascript:`、`data:`、`file:`。
5. 长链接 host 为 `localhost`。
6. 长链接 host 解析到内网 IP。
7. 短码不存在。
8. 短码命中已禁用记录。
9. 短码命中过期记录。
10. 短码生成多次冲突。
11. 并发创建同一 URL 时产生重复记录。
12. 高并发访问时点击数更新丢失。

## 验收标准

### 创建短链接

1. 已登录用户提交合法 `https` 长链接时，系统返回短链接信息。
2. 返回结果包含 `id`、`originalUrl`、`shortCode`、`shortUrl`、`createdAt`。
3. `shortCode` 全局唯一。
4. 同一用户重复提交同一个长链接时，返回同一个可用短链接。
5. 不同用户提交同一个长链接时，可以拥有各自的短链接。
6. 非法 URL 返回 `400 INVALID_URL`。
7. 非 `http` / `https` 协议返回 `400 UNSUPPORTED_URL_SCHEME`。
8. 内网地址或 localhost 返回 `400 URL_NOT_ALLOWED`。
9. 未登录创建时返回 `401 UNAUTHENTICATED`。

### 访问短链接

1. 访问存在且 active 的短码时，返回 `302`。
2. `Location` header 等于原始长链接。
3. 成功访问后 `click_count` 增加 1。
4. 访问不存在的短码时返回 404。
5. 访问已禁用短码时返回 410。
6. 访问已过期短码时返回 410。
7. 跳转接口不要求用户登录。

### 查询和管理

1. 已登录用户可以查询自己创建的短链接详情。
2. 已登录用户可以分页查询自己的短链接列表。
3. 用户不能查询或禁用他人的短链接。
4. 用户可以禁用自己创建的短链接。
5. 禁用后的短链接不再跳转。

### 数据和一致性

1. `short_code` 有唯一约束。
2. 并发创建时不会产生重复短码。
3. 并发访问时点击数不会明显丢失。
4. 所有创建、更新记录都有 `created_at` / `updated_at`。
5. 禁用操作不物理删除数据。

### 安全边界

1. 不允许跳转到 `javascript:`、`data:`、`file:` 等危险协议。
2. 不允许跳转到 localhost。
3. 不允许跳转到内网 IP 或保留地址段。
4. API 不返回其他用户的短链接数据。
5. 错误响应不暴露数据库细节、堆栈信息或内部实现。

## 待确认问题

1. 创建短链接是否一定需要登录，还是允许匿名创建？
2. 是否支持用户自定义短码？
3. 短链接默认是否永久有效？
4. 访问短链接时是否确定使用 `302`，还是希望使用 `301`？
5. 点击统计只要总数，还是需要访问日志和来源分析？
6. 短链接禁用后，访问端展示错误页面还是只返回状态码？
7. 是否需要创建频率限制，例如每用户每分钟最多创建 N 条？
8. 是否需要管理员后台管理所有短链接？

## 增量开发任务拆解

### 阶段 1：基础模型和通用能力

#### 任务 1：定义短链接数据模型和迁移

描述：建立 `short_links` 表及必要索引，为后续创建、查询、跳转和禁用能力提供数据基础。

输入：

1. 数据模型规格中的 `short_links` 字段定义。
2. 现有用户模型或认证系统中的用户 ID。
3. 当前项目使用的数据库迁移工具。

输出：

1. 新增 `short_links` 表。
2. 新增 `short_code` 唯一索引。
3. 新增 `user_id, created_at` 查询索引。
4. 新增时间字段和状态字段默认值。

验收条件：

1. 数据库迁移可以成功执行和回滚。
2. `short_code` 不能重复。
3. `status` 默认值为 `active`。
4. `click_count` 默认值为 `0`。
5. `created_at`、`updated_at` 能被正确写入。

#### 任务 2：实现统一错误响应

描述：定义短链接相关接口使用的统一错误格式，避免每个接口各自拼接错误响应。

输入：

1. 异常处理规格中的错误响应格式。
2. 错误码表。
3. 项目现有 API 错误处理方式。

输出：

1. 统一错误响应结构：`code` 和 `message`。
2. 短链接功能使用的错误码常量或枚举。
3. 对应 HTTP 状态码映射。

验收条件：

1. API 错误响应固定包含 `code` 和 `message`。
2. 未登录返回 `401 UNAUTHENTICATED`。
3. 无权限返回 `403 FORBIDDEN`。
4. 业务错误不会暴露数据库细节、堆栈信息或内部实现。

#### 任务 3：实现 URL 校验和归一化

描述：实现长链接输入校验，保证只接受安全、明确的 `http` / `https` URL。

输入：

1. 用户提交的 `url` 字符串。
2. 安全边界要求。
3. 允许协议列表：`http`、`https`。

输出：

1. URL 校验函数。
2. URL 归一化函数。
3. 对非法 URL、危险协议、localhost、内网地址的错误分类。

验收条件：

1. 空字符串返回 `400 URL_REQUIRED`。
2. 非 URL 返回 `400 INVALID_URL`。
3. 缺少协议的 URL 返回 `400 INVALID_URL`。
4. `javascript:`、`data:`、`file:` 返回 `400 UNSUPPORTED_URL_SCHEME`。
5. `localhost` 和内网地址返回 `400 URL_NOT_ALLOWED`。
6. 合法 `http` / `https` URL 可以通过校验并生成 `normalized_url`。

#### 任务 4：实现短码生成能力

描述：实现系统自动生成短码的基础能力，并处理短码冲突重试。

输入：

1. 短码字符集配置。
2. 短码长度配置。
3. `short_links.short_code` 唯一约束。

输出：

1. 短码生成函数。
2. 短码格式校验规则。
3. 冲突重试机制。

验收条件：

1. 生成的短码长度符合配置。
2. 生成的短码只包含允许字符。
3. 短码冲突时会自动重新生成。
4. 达到最大重试次数后返回 `500 SHORT_CODE_GENERATION_FAILED`。

### 检查点：基础能力

1. 数据库迁移成功。
2. URL 校验单元测试通过。
3. 短码生成单元测试通过。
4. 错误响应格式在测试中保持一致。

### 阶段 2：创建短链接主流程

#### 任务 5：实现创建短链接接口

描述：实现 `POST /api/short-links`，让已登录用户提交长链接并获得短链接。

输入：

1. 已登录用户身份。
2. 请求体中的 `url`。
3. URL 校验和归一化能力。
4. 短码生成能力。

输出：

1. 新增短链接记录。
2. 返回短链接详情。
3. 返回完整 `shortUrl`。

验收条件：

1. 未登录请求返回 `401 UNAUTHENTICATED`。
2. 合法 `https` URL 创建成功。
3. 响应包含 `id`、`originalUrl`、`shortCode`、`shortUrl`、`expiresAt`、`createdAt`。
4. 非法 URL 按错误码表返回对应错误。
5. 新创建记录的 `status` 为 `active`，`click_count` 为 `0`。

#### 任务 6：实现同用户重复 URL 复用

描述：同一用户重复提交相同归一化 URL 时，返回已有 active 短链接，避免重复创建。

输入：

1. 已登录用户身份。
2. `normalized_url`。
3. 已存在的 active 短链接记录。

输出：

1. 已存在短链接详情。
2. 不新增重复 active 记录。

验收条件：

1. 同一用户重复提交同一 URL 时返回相同 `id` 和 `shortCode`。
2. 同一用户重复提交不会新增第二条 active 记录。
3. 不同用户提交同一 URL 时可以生成各自的短链接。
4. 并发创建同一 URL 时不会产生重复 active 记录。

### 检查点：创建流程

1. 创建接口集成测试通过。
2. 重复提交测试通过。
3. 并发创建测试通过或已有数据库约束兜底。

### 阶段 3：访问跳转主流程

#### 任务 7：实现短链接跳转接口

描述：实现 `GET /{shortCode}`，访问有效短码时跳转到原始链接。

输入：

1. URL 路径中的 `shortCode`。
2. `short_links` 中 active 且未过期的记录。

输出：

1. `302 Redirect` 响应。
2. `Location` header 指向 `original_url`。

验收条件：

1. 存在且 active 的短码返回 `302`。
2. `Location` header 等于原始长链接。
3. 跳转接口不要求用户登录。
4. 短码格式非法时返回明确错误。
5. 短码不存在时返回 `404 SHORT_LINK_NOT_FOUND`。

#### 任务 8：实现点击次数递增

描述：每次有效短链接跳转时，可靠递增 `click_count`。

输入：

1. 有效短链接记录。
2. 跳转请求。

输出：

1. `click_count` 增加 1。
2. 跳转行为不受计数逻辑明显影响。

验收条件：

1. 每次成功跳转后 `click_count` 增加 1。
2. 不存在、禁用、过期的短链不会增加 `click_count`。
3. 并发访问时点击数不会明显丢失。
4. 计数失败时不暴露内部错误细节。

#### 任务 9：实现禁用和过期访问处理

描述：访问禁用或过期短链接时，不再跳转，并返回符合规格的错误。

输入：

1. `shortCode`。
2. `status` 字段。
3. `expires_at` 字段。

输出：

1. 禁用短链错误响应。
2. 过期短链错误响应。

验收条件：

1. `status = disabled` 的短链返回 `410 SHORT_LINK_DISABLED`。
2. `expires_at` 早于当前时间的短链返回 `410 SHORT_LINK_EXPIRED`。
3. 禁用或过期短链不会返回 `Location` header。
4. 禁用或过期短链不会增加 `click_count`。

### 检查点：跳转流程

1. 有效短链可以端到端跳转。
2. 点击次数递增测试通过。
3. 不存在、禁用、过期三类访问异常测试通过。

### 阶段 4：查询和管理

#### 任务 10：实现短链接详情查询接口

描述：实现 `GET /api/short-links/{id}`，让用户查看自己创建的短链接详情。

输入：

1. 已登录用户身份。
2. 路径参数 `id`。
3. `short_links` 记录。

输出：

1. 当前用户有权访问的短链接详情。
2. 对无权限或不存在记录的错误响应。

验收条件：

1. 已登录用户可以查询自己的短链接详情。
2. 响应包含 `id`、`originalUrl`、`shortCode`、`shortUrl`、`status`、`clickCount`、`expiresAt`、`createdAt`、`updatedAt`。
3. 查询他人短链接返回 `403 FORBIDDEN` 或按项目安全策略返回 404。
4. 未登录返回 `401 UNAUTHENTICATED`。

#### 任务 11：实现短链接列表查询接口

描述：实现 `GET /api/short-links?page=1&pageSize=20`，让用户分页查看自己创建的短链接。

输入：

1. 已登录用户身份。
2. 分页参数 `page`、`pageSize`。
3. 当前用户的短链接记录。

输出：

1. 分页列表。
2. 总数。
3. 当前页码和每页数量。

验收条件：

1. 已登录用户只能看到自己创建的短链接。
2. 默认按 `created_at` 倒序返回。
3. 响应包含 `items`、`page`、`pageSize`、`total`。
4. 分页参数非法时返回明确的 400 错误。
5. 未登录返回 `401 UNAUTHENTICATED`。

#### 任务 12：实现短链接禁用接口

描述：实现 `PATCH /api/short-links/{id}`，支持用户禁用自己创建的短链接。

输入：

1. 已登录用户身份。
2. 路径参数 `id`。
3. 请求体 `{ "status": "disabled" }`。

输出：

1. 更新后的短链接状态。
2. 后续访问该短链时不再跳转。

验收条件：

1. 用户可以禁用自己创建的短链接。
2. 用户不能禁用他人的短链接。
3. 禁用成功后返回 `id`、`status`、`updatedAt`。
4. 禁用后的短链接访问返回 `410 SHORT_LINK_DISABLED`。
5. MVP 不物理删除短链接记录。
